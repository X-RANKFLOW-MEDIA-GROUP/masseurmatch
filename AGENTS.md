# AGENTS.md — Guidance for automated coding agents

Purpose
- Provide concise, actionable guidance an AI coding agent can use to build, test, and modify this repo.
- Link to authoritative docs; avoid embedding large content.

Quick start (use these exactly)
- Install: `pnpm install`
- Dev server: `pnpm dev` (Next.js on port 5000)
- Build: `pnpm build` (runs `pnpm run clean:next` then `next build`)
- Start (prod-like): `pnpm start`
- Full checks: `pnpm run check-all` (lint + typecheck + tests + validations)

Conventions agents must follow
- This repo uses `pnpm` as the package manager (packageManager: pnpm@10.32.1). Use `pnpm` for installs and scripts.
- Do not attempt to run Prisma migrations or generate a Prisma client. `prisma/schema.prisma` is an archived reference only — the runtime uses Supabase.
- Respect environment variable conventions: `NEXT_PUBLIC_` for public values, non-prefixed for server-only secrets. See [.env.example](.env.example).
- Use `tsc --noEmit -p tsconfig.typecheck.json` (script: `typecheck`) for type-only checks.
- Lint with `pnpm run lint` (ESLint flat config in `eslint.config.mjs`).

Common pitfalls
- Missing `MM_SESSION_SECRET` will break server sessions in production — treat it as required for server runs.
- Stripe price IDs, webhook secrets, and Resend API keys are required for billing/email end-to-end tests; mock or skip when running isolated unit tests.
- The canonical DB source is Supabase; do not introduce Prisma runtime usage.

What agents should modify (and how)
- Short, focused changes only. Prefer editing or adding tests, small refactors, or docs.
- When adding or changing scripts, update [package.json](package.json) and include a one-line justification in the commit message.
- When touching infra/secret-related files, list required env vars and avoid committing secrets.
